# Streaming Admin Web運用コンソール

Coreを起動したあと、リポジトリ直下で次を実行し、
<http://127.0.0.1:8780> をブラウザで開く。

```bash
.venv/bin/python gui/yura-streaming-admin/server.py
```

待受ポートは `--port` で変更できる。Core APIの接続先、トークン、タイムアウト、操作者名は
`AI_LIVER_ADMIN_API_URL`、`AI_LIVER_ADMIN_API_TOKEN`、
`AI_LIVER_ADMIN_API_TIMEOUT`、`AI_LIVER_ADMIN_OPERATOR` で指定する。

Streaming Admin は配信制御を行うCore／UseCase／Adapterとは分離し、状態表示、操作要求、
確認ダイアログ、診断情報の保存を担当する。ブラウザは同一オリジンのローカルWebサーバーとのみ
通信し、Core APIトークンを受け取らない。画面からAdapterを直接呼び出さない。

## 状態取得と更新

- 初期表示と再接続時はRESTの`/api/v1/admin/console`を含む集約状態を取得する。
- 実行中の変化はSSEを契機にRESTで再同期する。OBSとYouTubeは手動更新も提供する。
- Coreが一つの状態遷移で複数イベントを発行した場合、Streaming Adminサーバーは短時間のイベントをまとめ、ブラウザへ1回の更新通知として送る。
- ブラウザは更新通知を受けると`/api/bootstrap`を1回取得する。SSEのheartbeatだけでは再取得しない。
- 配信セッションが存在しない間は、開始、Opening、Main、終了、コメント関連のセッション配下APIを取得しない。
- 状態更新時は画面全体のDOMを作り直さない。既存要素のテキスト、属性、必要な一覧だけを差分更新する。
- サービスカードとその操作ボタンはDOMを維持し、状態値だけを更新する。これにより更新中のホバー、フォーカス、クリック状態を失わない。
- テーブルや選択肢は内容の署名が変化した場合だけ差し替える。
- Service statusは`update_mode`、`last_updated_at`、`freshness`を持つ。鮮度は取得時刻と設定されたstale閾値からCore側で算出する。

## Adapter capabilityと人間操作

`streaming_capabilities`は公開開始・終了の自動実行、Studio URL、状態確認、確認要否を表す。
UIはAdapter名から人間操作を推測しない。Google/Fake control adapterはいずれも公開遷移を
自動実行でき、Fake構成ではStudio操作を要求しない。将来の手動Adapterは同じDTOで
`operator_action`を返す。

`operator_action`は`none`、`youtube_start_required`、`youtube_stop_required`、
`authentication_required`、`obs_confirmation_required`、`recovery_decision_required`を扱い、
状態は`not_required`、`waiting`、`acknowledged`、`completed`、`expired`とする。

## 診断とログ

Coreはファイルログの有効状態に関係なく、イベント、状態遷移、管理操作、失敗を最大500件の
リングバッファへ保持する。診断スナップショットにはruntime、Adapter、直近イベント、エラー、
設定要約を含める。画面の「表示をクリア」はブラウザ上の表示のみを消去し、Coreのリングバッファや
ファイルを変更しない。

実行時設定APIはログレベル、ファイル出力、保存先、ローテーション、リングバッファ件数、
OBS／YouTube更新間隔、stale閾値、操作待ちダイアログを受け付ける。設定変更は監査イベントに
記録され、配信ライフサイクルを停止しない。

## 画面コンセプト

Streaming Adminは「海上に浮かぶ管制塔」をモチーフとする。画面名「配信の灯台」は維持し、
配信・OBS・YouTube・コメント・進行状況を監視して必要な指示を出す海上管制塔として位置づける。
主要パネルは半透明のすりガラス表現とし、背景が透けても文字のコントラストを維持する。

### 海上・海中背景

海面の初期基準位置は`--sea-base: 40vh`とする。広い画面の概要では、上部状態カード左側の空間へ
灯台と海面が収まるよう配置する。灯台は海面へ追従しながら数pxの上下動と小さな傾きを繰り返す。

海面は円弧を並べず、辺に丸みを持たせた三角波を前後2層で描画し、異なる速度で水平移動させる。
スクロール時の上昇量はページ全体の進捗率ではなく、スクロールしたピクセル数へ固定係数を掛けて
算出する。画面高さやコンテンツ量にかかわらず同じスクロール量で同じ距離だけ海面が上昇し、上限は
画面上端付近とする。

海中は反復する斜線や大きな繰り返し模様を使用しない。海面直下の青緑から深海の暗い紺、最深部の
黒に近い色まで、縦方向へ連続的に暗くなるグラデーションで表現する。粒子は低い不透明度で補助的に
使用する。視差効果を減らす設定が有効な場合は波、灯台の漂流、海面移動を停止する。

## 画面構成

固定ヘッダーには次を配置する。

- 画面名
- 「概要」「コメント」「配信進行」「診断・ログ」のリンク型ナビゲーション
- Core接続状態
- 緊急停止
- 設定を開く3点ボタン

設定は通常タブに含めない。3点ボタンを押したとき、画面全体へオーバーレイを表示し、ログ設定、
OBS更新設定、YouTube更新設定、操作・診断設定を編集する。背景クリック、閉じるボタン、Escapeキーで
閉じられるようにする。

概要だけに次の共通運用カードを表示する。

1. 現在状態と次の推奨操作
   - 認証、準備、開始、通常終了のうち現在実行可能な主要操作を1件だけ強調表示する。
   - 広い画面ではカード自体を小型化して右側へ配置し、左側を灯台表示領域とする。
2. 配信設定
   - 配信枠と進行表を選択し、一覧再取得とYouTube認証を提供する。
3. 必要な人間操作
   - `operator_action`の案内とYouTube Studioへの導線を表示する。

コメント、配信進行、診断・ログでは上記3カードを表示せず、それぞれの画面固有コンテンツだけを
表示する。概要には加えてサービスカード、復旧操作、折りたたみ式の操作分担を表示する。
