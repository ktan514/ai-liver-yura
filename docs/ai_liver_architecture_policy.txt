AIライバー 活動駆動型モジュラーアーキテクチャ 方針メモ
Version: 0.1.0
作成日: 2026-07-01

============================================================
1. 目的
============================================================

本プロジェクトでは、LLMを用いてYouTube等で配信できるAIライバーを作成する。

目指すものは、単に外部入力へ回答するチャットボットではなく、
自律的に話題を出し、コメントに反応し、必要に応じて待機・観察・発話・表情制御を行う
「活動主体として振る舞うAIエージェント」である。

そのため、従来の「入力 → UseCase → 応答」のみを中心にした構成ではなく、
常時稼働するRuntimeを中心にした構成を採用する。


============================================================
2. 採用する基本アーキテクチャ
============================================================

採用方針:

    活動駆動型モジュラーエージェントアーキテクチャ
    Activity-Driven Modular Agent Architecture

構成要素:

    - Activity Runtime
    - Event Queue
    - Activity Manager
    - Action Planner
    - Channel Executors
    - Ports & Adapters

クリーンアーキテクチャの考え方は完全には捨てない。
ただし、アプリケーションの中心を Controller / UseCase ではなく、
EventQueue / ActivityManager / RuntimeCoordinator に置く。

クリーンアーキテクチャから採用するもの:

    - 依存方向を内側に向ける
    - Domainを外部サービスから独立させる
    - LLM / TTS / OBS / YouTube / Live2D などをAdapter化する
    - 抽象Portを通して差し替え可能にする

クリーンアーキテクチャ単体では弱い点:

    - 常時稼働する処理の中心概念が弱い
    - 外部入力がない状態での自律行動を表しにくい
    - 発話・表情・観察・記憶整理などの並行処理を扱いにくい
    - 1イベント1応答の逐次処理に寄りやすい


============================================================
3. 中心概念
============================================================

本システムでは、次の3つを明確に分ける。

------------------------------------------------------------
3.1 Event
------------------------------------------------------------

Eventは「起きたこと」を表す。

例:

    - user_text
    - youtube_comment
    - user_speech
    - camera_frame
    - silence_timeout
    - speech_finished
    - curiosity_peak
    - trend_updated
    - action_failed

外部入力だけでなく、内部状態の変化や実行結果もEventとして扱う。

------------------------------------------------------------
3.2 Activity
------------------------------------------------------------

Activityは「継続する目的」を表す。

例:

    - conversation_with_user
    - autonomous_talk
    - listening_mode
    - stimulus_reaction
    - curiosity_research
    - topic_exploration
    - external_trend_watch
    - body_expression_loop
    - idle_observation

Activityは active / waiting / suspended / completed / canceled などの状態を持つ。

------------------------------------------------------------
3.3 Action
------------------------------------------------------------

Actionは「今この瞬間に実行する具体的な命令」を表す。

例:

    - speak
    - ask
    - react
    - listen
    - observe
    - move
    - change_expression
    - update_subtitle
    - seek_information

Activityが目的、Actionが具体的な実行単位である。


============================================================
4. 全体処理フロー
============================================================

基本フロー:

    Input Receivers
        ↓
    AgentEvent
        ↓
    EventQueue
        ↓
    EventFilter / Prioritizer
        ↓
    ActivityManager
        ↓
    Deliberation Dispatcher
        ↓
    ActionPlanner
        ↓
    Channel Executors
        ↓
    Followup Event

入力がない場合も、内部状態やタイムアウトからEventを生成し、
Runtimeが次のActivityを進める。

例: 自律発話

    silence_timeout / curiosity_peak
        ↓
    autonomous_talk Activity 起動
        ↓
    TopicEngineで話題選定
        ↓
    speak Action 実行
        ↓
    speech_finished Event
        ↓
    次の話題継続 / 返答待ち / idleへ遷移

例: コメント反応

    YouTube comment received
        ↓
    AgentEvent(youtube_comment)
        ↓
    EventQueue
        ↓
    conversation_with_user Activityを前面化
        ↓
    応答方針を決定
        ↓
    speak + expression + subtitle Actionを実行

USER_TEXTは通常Eventより強い割り込み契機として扱う。
受理時点で進行中の割り込み可能なAUTONOMOUS_TALKをSUSPENDEDへ退避し、
対応するCONVERSATION_WITH_USERをforegroundへ予約する。
ExecutorはAction開始直前にActivityのACTIVE状態を再確認し、退避後の自律Actionを開始しない。
USER_TEXT受理時には未実行のAUTONOMOUS_TALKをPlannedActivityQueueから削除し、CANCELEDとして記録する。
Action生成中に割り込まれた場合も、ActionSchedulerへ渡す直前に状態を再確認して生成済みActionを破棄する。
すでに再生中の音声は初回実装では強制停止せず、完了後にユーザー応答を優先する。
発話・字幕・表情は共通のoutput_unit_idを持つ同一出力単位として扱う。
字幕と表情を反映してから音声を再生し、音声完了前に次の出力単位を開始しない。
出力単位および各Actionの開始・完了はoutput_unit_idとaction_idで追跡可能にする。
音声出力の優先順位は、ユーザー応答、通常の反応・挨拶、自律発話の順とする。
同じ優先順位では到着順を維持し、複数のユーザー応答を逆転させない。
再生中の音声は完了を待つが、待機中の自律発話は後着のユーザー応答より後に開始する。

5. モジュール差し替え方針
============================================================

各外部サービスはPortを通して扱い、具体実装はAdapterに閉じ込める。
Runtime / Domain / UseCaseは、特定製品名を知らない。

TTSの読み補正は音声合成AdapterやUseCaseへ直接埋め込まず、独立した辞書・補正コンポーネントとして扱う。
表示・会話履歴・長期記憶にはLLMの元表記を保持し、補正済みテキストはTTS入力にのみ使用する。
VOICEVOXのAudioQuery補正はProtocolで拡張可能にし、失敗時は未補正AudioQueryへ安全にフォールバックする。

知ってよい抽象:

    - LLMPort
    - TTSPort
    - CommentSourcePort
    - AvatarPort
    - StreamingPort
    - MemoryStorePort
    - AudioInputPort
    - VisionInputPort

知ってはいけない具体名:

    - OpenAI
    - Ollama
    - Gemini
    - VOICEVOX
    - AivisSpeech
    - YouTube
    - OBS
    - Live2D
    - VRM
    - PostgreSQL

差し替え例:

    ManualCommentSource → YouTubeCommentSource
    DummyTTS            → VOICEVOXTTS / AivisTTS
    DummyAvatar         → Live2DAvatar / VRMAvatar
    DummyStreaming      → OBSStreamingController
    OllamaLLM           → OpenAILLM / GeminiLLM / LLMRouter
    SQLiteMemoryStore   → PostgreSQLMemoryStore

7. MVP方針
============================================================

最小構成でも、RuntimeとPortは最初から用意する。
ただし、各AdapterはDummy実装でよい。

MVPの構成:

    PyQt6手入力
        ↓
    AgentEvent(user_text)
        ↓
    EventQueue
        ↓
    ActivityManager
        ↓
    LLMPort
        ↓
    TTSPort / DummyTTS
        ↓
    字幕表示 / DummyAvatar
        ↓
    speech_finished Event
        ↓
    Runtimeが次の状態へ遷移

MVPで実装するPort:

    - LLMPort
    - TTSPort
    - AvatarPort
    - StreamingPort
    - CommentSourcePort
    - MemoryStorePort

MVPで実装するAdapter:

    - OllamaLLM または DummyLLM
    - DummyTTS
    - DummyAvatar
    - PyQtStreamingView または DummyStreaming
    - ManualCommentSource
    - SQLiteMemoryStore または InMemoryStore

8. Activity優先度の基本方針
============================================================

Activityは固定順位だけでなく、状況に応じてスコアリングする。

基本式:

    final_priority
      = base_priority
      + event_bonus
      + urgency_bonus
      + user_focus_bonus
      + continuity_bonus
      - interruption_penalty
      - cost_penalty
      - stale_penalty

基本優先度の例:

    conversation_with_user : 高
    listening_mode         : 中〜高
    stimulus_reaction      : 中
    curiosity_research     : 中
    autonomous_talk        : 中
    topic_exploration      : 中〜低
    external_trend_watch   : 低
    body_expression_loop   : 低
    idle_observation       : 最低

ユーザーコメントや明示入力は強く優先する。
ただし、すべての刺激を必ず処理するのではなく、重要度に応じて処理・保留・破棄する。


============================================================
9. 配信向けの質問モデル
============================================================

配信では、質問したあとに完全停止して返答を待つ設計は避ける。

旧モデル:

    ask
        ↓
    listening_mode
        ↓
    返答が来るまで発話抑制

新モデル:

    ask
        ↓
    OpenPromptとして登録
        ↓
    foreground activityは進行継続
        ↓
    後から来たコメントをOpenPromptに関連付けて拾う

OpenPromptの例:

    - 今日どう？
    - みんなどっち派？
    - このゲーム音量大丈夫？

配信では、質問と待機を分離する。


============================================================
10. 思考レベルの方針
============================================================

すべてのイベントに重いLLM推論を使わない。

思考レベルを分ける。

    reactive_direct:
        定型反応、短い相槌、軽い表情変化

    neutral_only:
        軽い通常応答

    balanced:
        中立 + 良心など、少し配慮が必要な応答

    full_deliberation:
        三脳や深い思考が必要な応答

    deliberation_loop:
        情報不足時に追加収集して再思考する応答

三脳は「意味・判断・態度」を決めるために使う。
口パク、視線、軽い身体揺れなどの高頻度制御には使わない。


============================================================
11. 出力チャネル設計
============================================================

ActionPlanは、複数チャネルの出力を持てるようにする。

出力チャネル:

    - speech
    - subtitle
    - expression
    - motion
    - avatar_state
    - streaming_control
    - memory_update

発話しながら表情や体を動かすため、実行はチャネル別Executorに分ける。

例:

    ActionPlan
      ├─ SpeechExecutor
      ├─ SubtitleExecutor
      ├─ ExpressionExecutor
      ├─ MotionExecutor
      └─ StreamingExecutor

Live2D / 3D の細かい動きは、LLMではなく連続制御ループ側で扱う。

============================================================
13. 現時点の決定事項
============================================================

- AIライバーはチャットボットではなく、活動主体のAIエージェントとして作る。
- アーキテクチャは「活動駆動型モジュラーエージェントアーキテクチャ」とする。
- クリーンアーキテクチャの依存分離は採用するが、中心はUseCaseではなくRuntimeに置く。
- Event / Activity / Action を明確に分ける。
- 外部入力だけでなく内部状態変化もEvent化する。
- 各外部サービスはPort & Adapterで差し替え可能にする。
- 最小構成でもPortは最初から作る。
- 最初はDummy実装でよい。
- 配信向けには「質問」と「待機」を分離し、OpenPrompt方式を採用する。
- すべての処理に重いLLMを使わず、思考レベルを切り替える。
- Live2D / 3D の高頻度制御はLLMではなく専用制御層で扱う。
